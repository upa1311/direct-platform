# Правила работы с проектом Direct

Этот файл обязателен для всех людей и автоматизированных агентов, работающих в репозитории `food-delivery-platform`.

## Источник бизнес-требований

- Перед любым изменением бизнес-логики прочитать относящиеся к нему документы в [`docs/`](docs/).
- Утверждённые решения находятся в [`docs/decision-log.md`](docs/decision-log.md).
- Нерешённые вопросы находятся в [`docs/open-questions.md`](docs/open-questions.md).
- Нельзя придумывать отсутствующие бизнес-правила или молча выбирать один из спорных вариантов.
- Обнаруженное противоречие сначала показать пользователю и зафиксировать в документации. Не разрешать его самостоятельно.
- Изменение документации синхронизировать с соответствующим решением: утверждённое решение — с журналом решений, нерешённое — со списком открытых вопросов и затронутыми предметными документами.

## Текущий этап

- Пользователь разрешил создать локальный кликабельный Next.js-прототип и установить необходимые зависимости.
- Прототип использует только тестовые данные и не должен выдавать демо-действия за реальные SMS, QR-платежи, звонки или финансовые операции.
- Нерешённые бизнес-переходы допускаются в интерфейсе только как демонстрационные действия с явной пометкой «Требует бизнес-решения».
- Не создавать базу данных и не устанавливать Docker или PostgreSQL без отдельного разрешения.
- Не выполнять `git commit` без прямого разрешения пользователя.

## Windows

- В PowerShell использовать `npm.cmd` вместо `npm`.
- В PowerShell использовать `npx.cmd` вместо `npx`.
- Не изменять PowerShell Execution Policy.
- Не изменять системные настройки Windows без прямого указания пользователя.

## Архитектура и код

- Начинать с модульного монолита; не создавать микросервисы без доказанной необходимости.
- Использовать TypeScript в строгом режиме (`strict`).
- Пользовательский интерфейс писать на русском языке.
- Имена файлов, переменных, типов и функций писать на английском языке.
- Деньги хранить целыми значениями в минимальных единицах валюты вместе с `currencyCode`.
- Не зашивать изменяемые бизнес-параметры в UI-компоненты будущей рабочей системы; тестовые данные прототипа держать отдельно от представления.
- Защищённые данные фильтровать на сервере, а не только скрывать в интерфейсе.
- Старые заказы должны хранить неизменяемые снимки товарных и финансовых условий.
- Финансовые операции нельзя удалять или изменять бесследно; исправления оформляются отдельными прослеживаемыми записями.
- Ежедневные изменения ресторанов, меню, цен, доступности и публикации не должны требовать изменения исходного кода в рабочей системе.

## Работа с Git

- Рабочая ветка проекта — `main`, пока пользователь не задаст иную схему.
- Перед изменениями проверять `git status` и сохранять несвязанные пользовательские изменения.
- Не переписывать историю и не выполнять разрушительные команды без прямого разрешения.
