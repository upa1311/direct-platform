# Restaurant Accounting V1 — DONE

Двусторонний учёт обязательств между Direct и ресторанами: признание из
неизменяемых финансовых снимков заказов, ресторанная сверка (read-only) и
административная фиксация внешних расчётов с append-only аудитом. Реального
движения денег система не выполняет.

Документ описывает текущую реализацию. Внутренние технические имена приводятся
для точности; пользовательский интерфейс остаётся на русском без сырых enum.

## 1. Источники финансовых данных

- Единственный источник сумм заказа — неизменяемый `order.financials`
  (`FinancialSnapshot`).
- Старые заказы **не пересчитываются** по текущему меню, тарифам, комиссии или
  акциям: изменение настроек не меняет исторический отчёт.
- Прежний `SettlementEntry` сохранён **только для совместимости** и питает
  показатель «Ожидает расчёта по журналу комиссий».

## 2. Ресторанный интерфейс — `/restaurant/settlements`

Read-only, ресторан ничего не мутирует.

- Переключатель периода: Сегодня / 7 дней / 30 дней / Всё время (границы —
  календарные даты часового пояса ресторана, корректны через DST).
- Завершённые заказы за период; оплаченные отмены («Требуют внимания») —
  отдельным блоком, не в основных итогах.
- Виды: «По заказам», «По дням» (сверка), «Обязательства» (журнал).
- Открытая позиция («Взаимные обязательства») — **всегда за всё время** и не
  зависит от выбранного периода.

## 3. Двусторонний журнал

- Направления: ресторан должен Direct / Direct должен ресторану.
- Внутренние типы: `PLATFORM_COMMISSION` (комиссия ресторана перед Direct) и
  `RESTAURANT_PAYOUT` (выплата Direct ресторану).
- **Identity обязательства — `(orderId, type)`**; `source` в identity не входит.
- Признание только при реальном завершении заказа (`DELIVERED` / `PICKED_UP`).
- `recognizedAt` берётся из реального `completedAt` заказа, не из времени вызова.
- Суммы: A (деньги собрал ресторан) → комиссия = `platformCommissionReceivableCents`;
  B (деньги собрал Direct) → выплата = `restaurantNetAfterPlatformCommissionCents`;
  смешанный snapshot → обе записи.

## 4. Legacy migration

- Старые `SettlementEntry` мигрируются в записи журнала один раз
  (`source = LEGACY_COMMISSION_SETTLEMENT`), идемпотентно по `legacySettlementId`.
- Snapshot- и legacy-запись **не создают два обязательства** одного
  `(orderId, type)`.
- Повторный `serialize → parse` не увеличивает баланс и не дублирует записи.

## 5. Админский интерфейс — `/admin/settlements`

- Все рестораны, включая архивные/остановленные (история не исчезает).
- All-time открытая позиция: ресторан должен Direct, Direct должен ресторану,
  чистая позиция, число открытых/закрытых.
- Журнал обязательств с фильтрами статуса и направления; история решений.
- Закрытие открытой записи: `SETTLED` доступен для обоих направлений; `WAIVED` —
  **только для комиссионного требования Direct** (`RESTAURANT_OWES_DIRECT` +
  `PLATFORM_COMMISSION`).
- Комментарий (основание) обязателен; внешняя ссылка необязательна.
- Мутация идёт через сериализованный provider (`runSerializedActionMutation`) —
  на самом свежем persisted state, с rebase между вкладками.
- Ровно **одно append-only audit-событие** на обязательство.
- При закрытии комиссии синхронизируется старый `SettlementEntry` того же
  `(restaurantId, orderId)`: SETTLED → PAID, WAIVED → WAIVED (новый не создаётся).
- После успеха — устойчивое подтверждение, которое остаётся видимым, даже когда
  строка уходит из фильтра «Открытые».

## 6. Денежная семантика

- Система фиксирует **обязательства** и **административное подтверждение**.
- Система **не** выполняет банковский перевод.
- Система **не** списывает деньги со счёта.
- Система **не** делает автоматический взаимозачёт.
- Статус `SETTLED` означает **зафиксированное внешнее исполнение**, а не
  проведённую платформой транзакцию.

## 7. Privacy

Бухгалтерские view-model (ресторанный журнал и админский экран) не содержат:

- имя клиента, телефон, адрес;
- код выдачи (pickup code);
- комментарии к приготовлению (cooking comments);
- внутренние `orderId` / `entryId` / `legacySettlementId` в пользовательских
  таблицах и подтверждениях.

Показывается только публичный номер заказа (или «Старое начисление», если заказа
уже нет). Сырые enum переводятся в русские подписи через label-maps.

## 8. Проверенные инварианты (тестовые гарантии)

- Неизменяемые snapshots: изменение меню/тарифов/комиссии не меняет отчёт.
- Часовой пояс и DST: границы периодов по календарным датам ресторана.
- Только завершённые (`DELIVERED`/`PICKED_UP`) входят в итоги.
- Оплаченные отмены отделены от завершённых сумм.
- Сумма дней = общая summary за период.
- Двусторонность направлений (комиссия vs выплата).
- Identity `(orderId, type)` и дедупликация snapshot/legacy.
- `serialize/parse` не удваивает записи/баланс.
- Legacy migration идемпотентна.
- Fail-same-state на неуспешных действиях.
- Одно resolution event на обязательство (fail-closed по статусу и по событию).
- Ограничения `WAIVED` (только комиссия Direct).
- Provider rebase (мутация на свежем state, повтор не создаёт второе событие).
- Read-only selectors не мутируют state.

## 9. Известные ограничения V1

- Прототип на `localStorage`, не production-база данных.
- Нет реальной банковской интеграции.
- Нет частичной оплаты.
- Нет массового закрытия обязательств.
- Нет автоматического взаимозачёта.
- Нет payout-batch.
- Нет CSV/PDF export.
- Нет invoice / налогового учёта.
- Нет refund-workflow внутри accounting-модуля.
- Browser acceptance — не полноценный production E2E.

## 10. Следующий возможный трек

**RESTAURANT STATEMENTS / EXPORTS** (в этом релизе не реализуется):

- периодические statements;
- opening / closing position;
- CSV / PDF;
- payout batches.
