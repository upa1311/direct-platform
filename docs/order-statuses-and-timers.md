# Direct: статусы заказов и таймеры

## Назначение

Документ разделяет:

- явно утверждённые пользовательские этапы;
- подтверждённые события и таймеры;
- финансовые состояния;
- состояния публикации контента;
- ещё не утверждённые системные enum-имена и переходы.

Русские подписи из ТЗ не следует автоматически считать окончательными именами enum.

## Общие ограничения

- Клиент видит статусы и ориентировочное время заказа.
- Администратор видит временную шкалу и все таймеры.
- Один водитель Direct может иметь только один активный заказ.
- Один заказ должен атомарно приниматься только одним водителем.
- Один заказ содержит товары только одного ресторана.

## PLATFORM_DRIVER: подтверждённые события

ТЗ не задаёт полный исчерпывающий список системных статусов PLATFORM_DRIVER. Явно зафиксированы следующие события и отображаемые состояния:

| Событие | Утверждённый результат |
| --- | --- |
| Клиент отправил заказ | Ресторан должен проверить наличие и состав. Название системного статуса не задано. |
| Ресторан подтвердил состав | Состав и сумма могут быть зафиксированы для онлайн-оплаты. Название статуса не задано. |
| Ресторан предложил замену | Запускается 3-минутный таймер ответа клиента. |
| Клиент принял замену | Состав и сумма фиксируются; далее предоставляется онлайн-оплата. |
| Успешная онлайн-оплата | Заказ автоматически получает отображаемый статус «Готовится». |
| CashEnabled-водитель принял наличный заказ | Ресторан получает сигнал начинать готовить. |
| Ресторан нажал «Готово и упаковано» | Заказ считается приготовленным и упакованным; чек вложен; наклейка печатается; водитель и клиент уведомляются. |
| Водитель принял предложение | Ему открываются полный адрес и контакт клиента. |
| Заказ завершён | Доступ к телефонным номерам должен быть ограничен; точная политика не утверждена. |

Для PLATFORM_DRIVER не утверждены точные названия и полный порядок состояний наподобие «водитель едет в ресторан», «забрал заказ», «у двери», «доставлено». Их нельзя добавлять как утверждённую модель только по аналогии с другими режимами.

## Согласование и оплата онлайн

### Замена

- При предложенной замене клиенту даётся 3 минуты.
- Клиент может принять замену или отменить весь заказ.
- Если ответа нет 3 минуты, заказ полностью отменяется.
- Деньги до этого не списываются.

### Оплата онлайн

- После согласования состава и суммы клиенту даётся 5 минут на онлайн-оплату.
- До подтверждённой оплаты ресторан не начинает готовить.
- После успешной оплаты отображаемый статус автоматически становится «Готовится».

ТЗ не определяет действие по истечении 5 минут без оплаты. Автоматическая отмена, повторное выставление платёжной ссылки, продление или отдельный статус не утверждены.

## Таймер реакции ресторана

Для нового заказа утверждена шкала:

| Отсчёт | Действие |
| --- | --- |
| 00:00 | Новый заказ; громкий повторяющийся сигнал, полноэкранное и push-уведомление, обратный таймер. |
| 02:00 | Повторное уведомление ресторану. |
| 05:00 | Предупреждение администратору. |
| 08:00 | Автоматическая отмена, если администратор не продлил время. |

Сигнал прекращается только после явного действия сотрудника.

ТЗ не уточняет:

- какое именно действие считается достаточным для прекращения сигнала;
- на сколько администратор может продлить время;
- сколько продлений допустимо;
- какой статус присваивается при продлении;
- одинаков ли этот таймер для всех трёх режимов получения.

## Время приготовления

- При принятии ресторан подтверждает время приготовления.
- Интерфейс может показывать кнопку вида «Принять — готово через 25 минут».
- Ресторан может изменить время приготовления.
- ADMIN RESTAURANT MANAGEMENT требует настраиваемое время приготовления по умолчанию и время последнего заказа.
- После успешной онлайн-оплаты отдельная кнопка «Начать готовить» для обычного заказа не требуется.

Число 25 минут приведено как вид кнопки, а не как глобально утверждённое значение времени приготовления.

## Поиск водителя PLATFORM_DRIVER

Настраиваемый параметр по умолчанию:

    driverSearchLeadTimeMinutes = 10

Поиск оплаченного онлайн-заказа начинается за 10 минут до подтверждённого времени готовности, если администратор не запустил его раньше.

### Волны поиска

| Время от начала поиска | Охват |
| --- | --- |
| 0–30 секунд | Подходящие водители в зоне ресторана. |
| 30–60 секунд | Дополнительно подходящие водители соседних зон. |
| После 60 секунд | Администратор получает предупреждение. |

Заказ одновременно видят все подходящие водители текущей волны. Первый водитель, чьё принятие успешно зафиксировано атомарно, получает заказ.

Администратор может:

- начать поиск раньше;
- назначить водителя вручную;
- расширить поиск;
- отменить заказ в исключительном случае.

ТЗ не задаёт:

- длительность предложения для конкретного водителя;
- окончательный радиус после ручного расширения;
- автоматическую отмену после 60 секунд;
- точный статус заказа при отсутствии водителя.

### Особенность наличного заказа

- Проверенный cashEnabled-водитель должен быть найден до начала приготовления.
- Ресторан не начинает готовить, пока водитель не принял заказ.
- При принятии ресторан получает сигнал «Водитель найден — начинайте готовить».

## Готовность и упаковка

Действие «Готово и упаковано» атомарно в операционном смысле подтверждает:

- приготовление;
- формирование фискального чека;
- вложение чека;
- закрытие пакета;
- указание количества пакетов;
- формирование и отправку наклейки на печать;
- уведомление водителя;
- уведомление клиента.

Количество пакетов по умолчанию — 1, но сотрудник может изменить его до подтверждения.

Не утверждено отдельное системное имя статуса после этого действия.

## RESTAURANT_DELIVERY: утверждённые этапы

Ресторан обязан обновлять:

1. «Заказ принят».
2. «Заказ готовится».
3. «Заказ готов».
4. «Курьер ресторана выехал».
5. «Курьер скоро будет».
6. «Курьер у двери».
7. «Доставлено».

В первой версии эти этапы может менять сотрудник ресторана. Позже возможна отдельная роль RESTAURANT_COURIER.

ТЗ не задаёт:

- системные enum-имена;
- автоматические переходы;
- таймауты между этапами;
- критерий «скоро будет»;
- последствия пропуска этапа.

## PICKUP: утверждённые этапы

1. «Заказ принят».
2. «Заказ готовится».
3. «Заказ готов к выдаче».
4. «Заказ выдан».

Выдача подтверждается четырёхзначным кодом клиента.

Правила необоснованной неявки:

- первая — инцидент и предупреждение;
- вторая — блокировка оплаты при получении для самовывоза.

Не утверждены таймер неявки, срок ожидания, порядок признания причины уважительной и срок блокировки.

## Отмена и недействительность оплаты

Явно утверждённые случаи:

- отсутствие ответа клиента на замену 3 минуты — полная отмена без списания;
- отсутствие реакции ресторана 8 минут — автоматическая отмена, если администратор не продлил время;
- администратор может отменить заказ в исключительном случае;
- клиент может отменить весь заказ вместо принятия замены;
- если сумма еды наличного заказа упала ниже порога до начала приготовления, наличный способ недействителен; клиент должен изменить корзину, перейти на онлайн-оплату или отменить заказ.

Не утверждены:

- общий перечень причин отмены;
- возможность и правила отмены после начала приготовления;
- финансовые последствия поздней отмены;
- частичная отмена;
- ответственность и компенсация;
- действие при истечении 5 минут онлайн-оплаты.

## Платёжные и расчётные состояния

Финансовый журнал должен различать:

- начислено;
- доступно к выплате;
- выплата создана;
- выплата обрабатывается;
- выплата подтверждена;
- выплата не удалась;
- задолженность;
- погашение;
- взаимозачёт;
- возврат;
- административная корректировка.

Администратор видит статус оплаты и статус выплаты.

Эти понятия не являются утверждённым конечным автоматом заказа. Точные enum-имена, переходы, повторные попытки и терминальные состояния не заданы.

## Состояния контента — не статусы заказа

ADMIN RESTAURANT MANAGEMENT вводит состояния управляемого контента:

- DRAFT;
- PENDING_REVIEW;
- PUBLISHED;
- HIDDEN;
- ARCHIVED.

Они относятся к публикации ресторана, меню, промо- и другого контента, а не к жизненному циклу заказа.

Администратор должен иметь предварительный просмотр до публикации. Политика модерации новых позиций, описаний и изображений настраивается. Ресторанные пользователи могут немедленно менять доступность и цены.

Публикация, скрытие или архивирование сущности не меняет снимки уже созданных заказов.

## Аудит переходов и таймеров

Администратор должен видеть временную шкалу и все таймеры заказа.

Каждое изменение конфигурации или ручная корректировка должна сохранять:

- actor;
- timestamp;
- entity;
- previous value;
- new value;
- reason, когда требуется.

Для ручного изменения финансовых сумм причина обязательна по смыслу отдельного требования о финансовых корректировках.

ТЗ не задаёт, какие нефинансовые переходы требуют обязательной причины. Это должно быть определено отдельно.

## Нерешённые вопросы

- Полный автомат и enum-имена PLATFORM_DRIVER.
- Действие после 5 минут без онлайн-оплаты.
- Правила продления 8-минутного таймера ресторану.
- Таймеры и автоматизация RESTAURANT_DELIVERY.
- Таймер ожидания и правила неявки PICKUP.
- Политика отмены после начала приготовления.
- Правила возвратов и компенсаций.
- Срок ограничения контактов после завершения заказа.
